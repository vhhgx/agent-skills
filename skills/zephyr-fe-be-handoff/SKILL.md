# Zephyr Frontend Backend Handoff

## 目标

在前端完成页面、交互、Mock、类型、接口调用或状态管理后，生成后端可直接开发的交接文档。

交接文档必须明确：前端已完成内容、前端实际调用或依赖、当前后端可复用能力、后端需要新增或修改的能力、Mock 与后端现状差异、联调和 Mock 关闭条件。

本 Skill 不负责全系统 API 治理、全量 OpenAPI、全量错误码、数据库迁移设计、通用认证权限架构、功能规格与项目计划、部署架构和服务拆分设计。

## Reference 读取规则

按任务读取需要的 reference；不得一次性读取全部 reference。

| 任务场景 | 必须读取 |
| --- | --- |
| 需要确定交接文档章节、各章节写什么 | `references/handoff-document-template.md` |
| 需要编写单个接口的 Method、Path、参数、响应字段、权限、错误处理、Checklist | `references/interface-detail-template.md` |
| 需要分析 Mock、字段映射、状态映射、行为映射 | `references/mock-and-mapping.md` |
| 需要拆后端任务、写联调与验收标准、做生成前自检 | `references/tasks-acceptance-checklist.md` |

## 输入与前置条件

生成交接文档前，必须取得当前任务相关的工作区分析结果。分析结果可以来自此前已执行的工作区证据分析，也可以由本 Skill 针对交接任务补充。

当前会话仅用于补充需求背景、已确认决策和用户指定约束，不得替代前端或后端工作区事实。

必须检查当前工作区变更：

```bash
git status
git diff
git diff --cached
```

无法获取 Git 变更时，交接文档必须记录原因，不得假装已完成 Diff 分析。

## 前端输入范围

按当前任务范围收集：页面与路由、交互、接口调用、状态管理、类型与校验、Mock、权限展示、错误处理、测试。

每个前端结论必须定位到文件路径与具体组件、函数、Hook、Store、类型、路由、配置项或测试。

## 后端核对范围

后端核对只围绕前端已表达的能力进行，不做无边界扫描。

按当前任务范围确认：接口入口、业务实现、数据访问、请求响应、配置与依赖、已有约束、异步能力、测试。

每个后端结论必须定位到文件路径与类、方法、路由、配置项、表、迁移或测试。

## 交接项分类

| 状态 | 含义 |
| --- | --- |
| 复用 | 后端已有能力，前端可直接接入或仅需调整调用 |
| 修改 | 后端已有能力，但字段、行为、状态、错误处理或返回结果不足 |
| 新增 | 当前检索范围内未发现对应能力，且前端需求已明确 |
| 前端修正 | 后端已有能力，问题来自前端调用、字段映射、状态处理、Mock 或错误处理 |
| 待确认 | 当前工作区和已确认需求无法判断是否需要开发或如何实现 |

“未发现”仅表示当前检索范围内未找到，不等于系统绝对不存在。禁止将“前端尚未找到后端实现”直接写成“后端必须新增”。

## 交接分析流程

1. 确定交接边界：功能名称、页面或路由范围、前端变更范围、后端模块范围、已确认规则、非本次范围。
2. 还原前端实际行为：页面入口 → 用户操作 → 前端状态变化 → 接口或 Mock 调用 → 返回数据消费 → 页面反馈。
3. 核对后端现状：前端调用或 Mock → 后端入口 → 业务实现 → 数据或外部依赖 → 返回或副作用。
4. 识别真实缺口：只有前端已调用或明确依赖，且后端无法满足已确认需求时，才产生后端交接项。
5. 排定实施顺序：优先解除 Mock 和主流程联调阻塞。
6. 按 Reference 读取规则加载需要的模板。
7. 输出交接文档并执行自检。

## 完成标准

交接文档必须满足：

- 基于当前工作区中的前端、后端、配置、类型、Mock 和 Git 变更事实。
- 明确前端已完成内容、后端现状和后端能力缺口。
- 明确区分复用、修改、新增、前端修正和待确认。
- 不将前端假设或 Mock 数据写成后端事实。
- 每个接口都包含当前可确认的 Method、Path、前端调用位置、用途、请求参数、响应字段、Mock 来源、权限或租户要求、错误处理要求、待确认问题和后端开发 Checklist。
- 无法从前端、后端、项目规范或已确认需求确认的内容，明确标记为待确认。
- Mock 与真实后端差异可用于指导实现、切换与联调。
- 后端任务按依赖顺序排列，可直接进入开发。
