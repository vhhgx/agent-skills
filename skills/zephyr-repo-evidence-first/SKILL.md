# Zephyr Repo Evidence First

## 执行标识

使用本 Skill 处理当前工作区任务时，最终回答的第一行必须写：

```text
依据 zephyr-repo-evidence-first：已完成当前任务相关代码、配置、调用链与同类实现的证据分析
```

如只完成部分分析，必须如实写：

```text
依据 zephyr-repo-evidence-first：已完成部分证据分析；未确认项见“待确认项”
```

禁止在未检索工作区、未形成证据摘要或仅依据通用经验时输出上述标识。

## 目标

确保 Agent 在处理现有项目时，先理解当前仓库的真实实现，再回答、设计或修改代码。

适用于回答当前项目实现、新功能开发、Bug 修复、重构、接口修改、数据库修改、前后端联调、第三方依赖集成、架构分析、技术方案评审、文档生成与审查。

核心原则：没有仓库证据，不得下项目结论；没有调用链证据，不得设计替代实现；无法确认时，必须明确标记“未确认”。

## Reference 读取规则

按任务读取需要的 reference；不得一次性读取全部 reference。

| 任务场景 | 必须读取 |
| --- | --- |
| 需要判断“证据是否足够”或为结论补充证据 | `references/evidence-requirements.md` |
| 任务涉及前端页面、路由、组件、API client、Mock、权限或状态 | `references/frontend-evidence.md` |
| 需要输出证据摘要、方案、实施前计划或最终结果 | `references/response-and-plan-format.md` |
| 需要审查回答是否违反证据优先原则 | `references/forbidden-and-checklist.md` |

## 强制规则

### 先分析，后回答或修改

涉及当前工作区的任何问题，必须先定位与任务直接相关的入口与调用方、业务实现、数据模型、配置与依赖、接口或事件定义、测试、同类已有实现、前后端关联代码。

不得只凭技术栈、文件名、目录名、依赖名称、常见项目结构、通用最佳实践或用户描述中的猜测推断当前项目实现。

### 禁止条件式猜测

未完成代码分析前，禁止用“如果项目已经有”“通常需要”“一般会有”“大概率”“可以考虑”“建议新建”“可能存在”等表达替代结论。

只有在工作区无法确认、且确实需要外部信息时，才能列为待确认项。

### 结论必须可回溯

所有关于当前项目的结论必须附带证据。证据至少定位到文件路径 + 类 / 函数 / 配置项 / 路由 / 类型 / 测试 / 调用点。

## 任务驱动分析流程

1. 提取任务边界：目标、影响面、关键词、预期结果。
2. 建立最小仓库地图：项目说明、构建文件、模块目录、配置文件、部署文件、测试目录。
3. 定位相关调用链：入口、路由或事件、业务实现、数据读写、类型或实体、配置依赖、错误处理、测试、同类功能。
4. 输出证据摘要：已确认现状、已确认缺口、待确认项。
5. 基于证据提出方案：复用、补齐、新增。
6. 实施前列计划：修改、新增、不修改，以及原因、影响和验证路径。
7. 完成修改后执行必要验证：构建、测试、接口或页面验证、配置校验、Git diff 审查、受影响调用链回归。

## 完成标准

本 Skill 执行成功，必须满足：

- 当前项目结论均可定位到代码、配置、测试、迁移或部署文件。
- 方案明确区分复用、补齐和新增。
- 未确认项与已确认结论分离。
- 实施前已说明调用链、影响范围和验证方式。
- 不存在基于惯例生成的平行实现。
- 人类开发者能够快速复查 Agent 的判断依据。
