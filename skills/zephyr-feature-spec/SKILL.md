# Zephyr Feature Spec

## 目标

将需求讨论、当前工作区实现和项目规范整理为功能规格与实施计划文档。

文档第一读者是人类开发者、负责人或 Reviewer；确认后可作为前端、后端、数据库和测试工作的共同依据。

本 Skill 的输出是“功能规格 + 实施计划”，不是泛泛需求说明，也不是只包含“新增页面、开发接口、补测试”的空任务列表。

## Reference 读取规则

按任务读取需要的 reference；不得一次性读取全部 reference。

| 任务场景 | 必须读取 |
| --- | --- |
| 需要确定文档输出目录、文件命名、序号规则 | `references/output-location-naming.md` |
| 需要生成完整功能规格文档章节 | `references/spec-document-template.md` |
| 需要定义范围、核心规则、影响面、接口边界、实施任务、验收标准 | `references/spec-content-rules.md` |
| 需要压缩表达、避免模糊词、做生成前自检 | `references/expression-and-checklist.md` |

## 前置条件

生成功能规格前，必须取得当前任务相关的工作区分析结果。该结果应覆盖相关代码、配置、依赖、调用链、测试和项目规范；当前会话仅用于补充需求背景与已确认决策，不得替代工作区事实。

如当前会话已存在可复用的工作区分析结果，应先检查其是否覆盖当前任务、是否包含可定位证据、以及相关文件是否已发生变更。满足条件时可直接复用；不满足时，必须补充或重新分析缺失部分。

可复用的工作区分析结果至少包含：当前实现、相关调用链、可复用模块、已确认缺口、待确认项，以及对应的文件路径、类、方法、组件、配置项或测试证据。

未完成相关分析时，不得将方案表述为当前项目结论。

## 任务执行流程

1. 识别功能边界：功能目标、用户角色、业务入口、影响范围、已确认决策、未确认决策。
2. 还原当前现状：入口、当前行为、调用链、数据来源、同类能力、当前限制、已有测试。
3. 定义本期范围：本期实现、非本期范围、后续阶段依赖、缺少确认而暂不进入本期的内容。
4. 定义核心规则：触发条件、处理动作、结果、异常行为、归属模块、是否需要人工确认。
5. 拆分前端、后端与数据影响。
6. 定义接口与交互边界。
7. 拆分实施任务，按依赖顺序排列。
8. 定义可观察、可测试的验收标准。
9. 列出待确认项，并与已确认事实隔离。
10. 按 Reference 读取规则加载需要的模板和检查清单。

## 工作区事实范围

工作区事实包括：代码、配置、依赖、路由、接口调用、类型定义、数据模型、迁移文件、测试、部署文件、同类实现。

工作区分析必须引用可定位证据。

| 类型 | 示例 |
| --- | --- |
| 代码 | 文件路径、类、方法、组件、Hook、Store |
| 配置 | YAML、环境变量、构建配置、部署配置 |
| 依赖 | Maven、Gradle、npm、pnpm 依赖声明 |
| 调用链 | 路由、API client、Controller、Service、Mapper |
| 数据 | Entity、DTO、Schema、迁移、SQL、查询 |
| 测试 | 测试文件、测试用例、Mock、断言 |
| 同类实现 | 已有页面、接口、模块、任务或事件处理 |

## 完成标准

本 Skill 成功执行时，必须满足：

- 已生成一个符合命名规则的功能规格文件。
- 输出目录遵循用户指定路径、项目规范或目录解析规则。
- 文档基于当前工作区事实，而非通用经验。
- 文档明确范围、规则、影响、计划、验收和待确认项。
- 已确认事实与待确认项严格分离。
- 人类读者可在短时间内完成首轮审阅。
- 后续开发者可据此继续细化接口、数据模型和实现任务。
