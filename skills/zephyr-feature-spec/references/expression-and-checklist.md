# 表达规范与生成前自检

## 保留定义，压缩解释

保留：接口路径、权限字符、状态值、字段名、实体名、事件名、错误码、配置项、任务名称。

压缩：背景说明、重复原因、常识性解释、不影响实现的业务描述。

## 禁止行为

- 用代码块包裹普通列表。
- 用“建议”“可以考虑”“通常”“一般”描述本期需求。
- 用一长段文字描述状态机、字段映射、权限关系或接口关系。
- 将未确认内容写成实现结论。
- 为了完整而重复现有工作区事实。
- 把完整 DTO、SQL、JSON 示例塞入正文。
- 将实施计划写成“新增前端、开发后端、测试”这种空任务。
- 按技术栈惯例补全未确认模块。

## 允许使用代码块的场景

仅限 SQL、JSON、YAML、TypeScript 或 Java 类型定义、可执行命令、配置片段、精确 Schema。

单个代码块超过 15 行时，应移到附录或单独设计文档。

## 文档边界

本 Skill 负责形成“功能规格与实施计划”。

文档可以包含范围、当前现状、业务规则、前端影响、后端影响、数据影响、接口边界、实施计划、验收标准、待确认项。

当接口字段、错误码、数据库字段、索引、迁移脚本或代码实现细节需要进一步明确时，可在后续阶段补充相应设计文档或实现说明。

## 生成前自检

- 是否已经完成当前任务相关的工作区分析。
- 是否所有“当前项目”结论都有文件、配置、调用链或测试依据。
- 是否区分已确认事实与待确认项。
- 是否明确本期范围和非本期范围。
- 是否避免使用“建议”“通常”“如果项目有”等模糊表达。
- 是否将规则写成表格、映射、状态机或短句。
- 是否只列出当前功能真正影响的前端、后端、数据模块。
- 是否每个实施任务都有明确对象、依赖和产出。
- 是否验收标准可执行、可观察。
- 是否已根据项目规范确定输出目录。
- 是否已检查当天文件序号并避免覆盖。
- 是否减少解释型段落、无意义代码块和无意义列表。
- 是否让人类可以快速审阅主要结论和实施边界。
