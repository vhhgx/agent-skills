# 功能规格内容规则

## 核心规则

只写可执行规则，不写解释型段落。

| 内容类型 | 推荐表达方式 |
| --- | --- |
| 映射关系 | 表格 |
| 状态迁移 | 状态迁移表 |
| 权限定义 | 权限码映射表 |
| 接口依赖 | 契约表 |
| 数据关系 | 实体关系表 |
| 处理顺序 | 有序步骤 |
| 边界条件 | 条件与结果表 |

规则必须尽量明确：触发条件、处理动作、结果、异常行为、归属模块、是否需要人工确认。

## 前端影响

| 页面/模块 | 变化 | 依赖 | 验收点 |
| --- | --- | --- | --- |
|  |  |  |  |

## 后端影响

| 模块 | 变化 | 依赖 | 验收点 |
| --- | --- | --- | --- |
|  |  |  |  |

后端影响可包括 Controller、Service、Repository、Mapper、任务、事件、消息、缓存、校验、异常处理、幂等、并发、外部服务调用、配置、定时任务、部署依赖。

## 数据影响

| 实体/表 | 变化 | 原因 | 迁移风险 |

只写当前阶段所需的数据影响。字段、索引、SQL 和迁移细节可在后续设计或开发阶段补充。

某层无直接影响时，写：无直接影响：<依据>。

## 接口与交互边界

功能规格中只定义接口级或事件级需求，不强制写完整 DTO 或 OpenAPI。

| 调用方 | 方法 | 路径/事件 | 用途 | 约束 | 状态 |
| --- | --- | --- | --- | --- | --- |
|  |  |  |  |  | 新增 / 修改 / 复用 |

字段、错误码、分页、幂等、并发和完整请求响应结构尚未明确时，标记：待补充契约。

## 实施任务

按依赖顺序拆分，不按“前端 / 后端”机械分组。

| 优先级 | 阶段 | 任务 | 依赖 | 产出 |
| --- | --- | --- | --- | --- |
| P0 | 基础能力 |  |  |  |
| P1 | 核心流程 |  |  |  |
| P2 | 完善能力 |  |  |  |

任务必须可独立验证，有明确依赖，不把多个不相关改动混入同一任务，不把“分析/开发/测试”写成没有对象的空任务。

## 验收标准

验收标准必须可观察、可测试。

| 场景 | 前置条件 | 操作 | 预期结果 |
| --- | --- | --- | --- |
|  |  |  |  |

至少覆盖正常主流程、核心失败路径、边界限制、数据一致性、前后端联动、回归影响范围。

禁止使用：功能正常、体验良好、符合预期。
