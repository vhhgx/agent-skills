# Zephyr Engineering Document Style

## 目标

生成、改写或审查高信息密度中文工程文档，使文档满足：人类可快速审阅，定义、规则和约束完整，后续 Agent 可继续分析、设计或编码。

核心原则：Human First, Agent Friendly。保留定义，压缩规则，删除无效解释。

本 Skill 只约束“文档如何写”，不规定某类文档必须有哪些章节，也不替代功能规格、接口交接或仓库证据分析。

## Reference 读取规则

按任务读取需要的 reference；不得一次性读取全部 reference。

| 任务场景 | 必须读取 |
| --- | --- |
| 判断代码块、列表、表格、标题如何使用 | `references/markdown-rules.md` |
| 编写权限、状态、字段、接口摘要、状态机、映射关系 | `references/structure-patterns.md` |
| 压缩中文表达、处理模糊词、控制篇幅 | `references/chinese-style.md` |
| 改写已有文档、审查文档质量、需要示例对照或最终自检 | `references/examples-and-checklist.md` |

## 执行流程

1. 确认文档类型、目标读者和输出形式。
2. 提取必须保留的定义：接口路径、权限字符、状态值、字段名、实体名、事件名、错误码、配置项。
3. 判断内容类型：定义、规则、解释。
4. 按 Reference 读取规则加载相关文件。
5. 将关系型内容改为表格、映射或状态迁移表。
6. 删除不影响决策和实现的解释、背景、重复标题和模糊措辞。
7. 将不确定内容标记为“待确认”，不得写成结论。
8. 输出前执行自检。

## 写作总原则

1. 信息密度优先。
2. 定义必须完整，规则必须明确，解释默认压缩。
3. 能一行表达的内容，不拆成列表。
4. 能表格表达的关系，不写成自然语言段落。
5. 能直接展示结构，不描述结构。
6. 不为了 Markdown 排版增加篇幅。
7. 不重复表达同一事实。
8. 不把推测写成结论。
9. 不使用博客、培训材料或 PRD 式铺垫代替工程事实。
10. 不确定内容必须明确标记为“待确认”。

## 完成标准

最终文档必须满足：

- 中文简洁、准确、可审阅。
- 定义完整且未被截断。
- 规则明确、可执行。
- 已确认结论与待确认项分离。
- 状态、权限、字段、接口等结构以合适形式直接展示。
- 代码块仅用于必须保留格式或可执行内容。
- 列表、表格、标题使用克制。
- 正文保留决策所需信息，不复制大段源代码或配置。
- 人类可快速完成首轮审阅。
- 后续 Agent 可准确理解约束并继续执行。
