# Skill: engineering-doc-density-writer

## 目标

生成高信息密度、适合人类审阅且便于 Agent 后续编码执行的工程文档。

适用于：

* 需求拆分文档
* 前后端交接文档
* 后端接口需求文档
* 架构设计文档
* 功能推进文档
* 数据模型说明
* 权限设计说明
* 状态机说明
* Agent 编码任务文档

核心原则：

> Human First, Agent Friendly.

文档首先要让人类快速审阅，其次让 Agent 能直接执行。

---

## 写作总原则

1. 信息密度优先。
2. 删除解释，保留定义。
3. 删除废话，保留规则。
4. 能一行说清，不拆成列表。
5. 能表格表达，不写自然语言段落。
6. 能结构化表达，不写流水账。
7. 不为了 Markdown 好看而增加篇幅。
8. 不使用博客、PRD、培训材料式写法。
9. 不写“众所周知”“为了实现”“我们需要”等空话。
10. 所有不确定内容必须标记为“待确认”。

---

## 内容分类规则

写文档时必须先判断内容类型。

### 1. Definition：定义

必须保留，并尽量结构化。

包括：

* 接口路径
* 权限字符
* 状态枚举
* 错误码
* 数据库字段
* DTO 字段
* 配置项
* 事件名
* Topic
* WebSocket 事件
* 任务类型
* 业务实体

推荐格式：

| 字段      | 说明   |
| ------- | ---- |
| account | 查看账号 |
| account | 创建账号 |

不要写成代码块。

---

### 2. Rule：规则

必须保留，但要压缩。

坏例子：

前端权限控制只用于用户体验，后端必须校验：

* 登录态
* 当前租户
* 资源归属
* 角色权限
* 敏感操作审计

好例子：

后端不得信任前端权限判断，所有业务接口必须校验认证、租户、资源归属、权限和审计。

---

### 3. Explanation：解释

默认删除。

例如：

“多租户用于支持多个客户、团队或业务线共用同一套系统，同时保证数据完全隔离。”

应改为：

多租户隔离对象：账号、会话、消息、联系人、代理、任务、模板、日志、监控、告警。

---

## Markdown 使用规则

### 禁止滥用代码块

除非内容必须被复制执行，否则不要使用三个反引号代码块。

禁止把以下内容放入代码块：

* 普通列表
* 权限字符说明
* 状态枚举说明
* 字段说明
* 页面字段
* 操作项
* 实施步骤
* 设计原则

优先使用表格或单行文本。

### 允许使用代码块的场景

仅限：

* SQL
* JSON
* YAML
* TypeScript interface
* OpenAPI
* Shell 命令
* 配置文件
* 可执行代码
* 精确 Schema

---

## 列表使用规则

禁止无意义拆列表。

坏例子：

需要校验：

* 权限
* 租户
* 频控
* 审计

好例子：

发送前校验：权限、租户、频控、文本审计。

只有满足以下条件才使用列表：

* 每一项都需要单独说明
* 每一项长度较长
* 每一项有独立状态、优先级或负责人
* 列表超过 5 项且不适合表格

---

## 表格优先规则

以下内容必须优先使用表格。

### 权限字符

| 权限字符    | 说明   |
| ------- | ---- |
| account | 查看账号 |
| account | 创建账号 |
| account | 编辑账号 |

### 状态枚举

| 状态            | 说明  |
| ------------- | --- |
| PENDING_LOGIN | 待登录 |
| PAIRING       | 配对中 |
| ONLINE        | 在线  |

### 状态机

| 当前状态          | 事件   | 下一状态         |
| ------------- | ---- | ------------ |
| PENDING_LOGIN | 请求配对 | PAIRING      |
| PAIRING       | 登录成功 | ONLINE       |
| ONLINE        | 连接断开 | OFFLINE      |
| OFFLINE       | 开始重连 | RECONNECTING |
| RECONNECTING  | 重连成功 | ONLINE       |

### 接口契约

| 方法   | 路径                      | 权限          | 说明   |
| ---- | ----------------------- | ----------- | ---- |
| GET  | /admin-api/system/users | system:user | 用户列表 |
| POST | /admin-api/system/users | system:user | 创建用户 |

### 页面字段

| 字段               | 说明     |
| ---------------- | ------ |
| lifecycleStatus  | 生命周期状态 |
| connectionStatus | 连接状态   |
| lastOnlineAt     | 最近在线时间 |

---

## 状态机写法

禁止流水账：

创建账号 -> PENDING_LOGIN
请求二维码 -> PAIRING
登录成功 -> ONLINE

必须改为状态迁移表：

| 当前状态          | 事件        | 下一状态          |
| ------------- | --------- | ------------- |
| 新建            | 创建账号      | PENDING_LOGIN |
| PENDING_LOGIN | 请求二维码/配对码 | PAIRING       |
| PAIRING       | 登录成功      | ONLINE        |
| ONLINE        | 连接断开      | OFFLINE       |

异常状态统一写：

| 当前状态 | 事件   | 下一状态            |
| ---- | ---- | --------------- |
| 任意状态 | 登录失效 | LOGIN_EXPIRED   |
| 任意状态 | 代理异常 | PROXY_ERROR     |
| 任意状态 | 风控异常 | RISK_CONTROLLED |

---

## 接口文档写法

接口必须包含：

* Method
* Path
* Permission
* Request
* Response
* Used By
* Backend Notes

推荐格式：

### GET /admin-api/system/users

| 项    | 内容                     |
| ---- | ---------------------- |
| 权限   | system:user            |
| 用途   | 查询当前租户用户列表             |
| 前端调用 | src/pages/system/users |
| 优先级  | P0                     |

Query:

| 字段       | 类型     | 必填 | 说明       |
| -------- | ------ | -- | -------- |
| page     | number | 否  | 页码       |
| pageSize | number | 否  | 每页数量     |
| keyword  | string | 否  | 用户名/昵称搜索 |

Response:

| 字段    | 类型     | 说明   |
| ----- | ------ | ---- |
| total | number | 总数   |
| list  | User[] | 用户列表 |

DTO、JSON 示例、OpenAPI 放到附录，不要塞在正文。

## 中文语法规则

1. 避免“进行”“实现”“相关”“能力”“支持”等空泛词。
2. “建议”必须改成“要求”“可选”“非本期”或“待确认”。
3. 每句话只表达一个事实。
4. 不写长句。
5. 不写“为了……需要……”。
6. 不写“主要是用于……”。
7. 不写“可以对……进行……”。
8. 动词优先使用：校验、返回、创建、更新、删除、过滤、记录、阻断、跳过。

坏例子：

系统需要支持对账号进行批量重连操作。

好例子：

支持批量重连账号。

坏例子：

该功能主要用于提升客服人员的回复效率。

好例子：

客服可通过快捷回复插入常用话术，插入后仍需人工发送。

---

## 模糊词处理

禁止直接使用：

* 建议
* 最好
* 可以考虑
* 需要的话
* 可能
* 尽量
* 相关
* 等等
* 一些
* 若干

必须替换为：

* 必须
* 可选
* 本期实现
* 后续实现
* 待确认
* 非本期范围

---

## 章节长度规则

1. 背景不超过全文 10%。
2. 单个章节超过 30 行必须拆分。
3. 单个代码块超过 15 行必须移到附录。
4. 连续列表超过 8 项，优先改表格。
5. 每个章节开头不要写空泛目标。
6. 标题下必须直接给定义、规则、表格或结论。

---

## 文档推荐结构

### 1. 结论

用 3～5 行说明本次文档要交付什么。

### 2. 范围

| 模块  | 本期范围      | 非本期范围 |
| --- | --------- | ----- |
| 多租户 | 数据隔离、租户切换 | 跨租户报表 |
| 权限  | 菜单权限、接口权限 | 字段级权限 |

### 3. 核心定义

包括状态、权限、实体、字段、错误码。

### 4. 业务规则

只写规则，不写解释。

### 5. 接口契约

给后端或 Agent 执行。

### 6. 数据模型

实体、字段、关系、索引、租户字段。

### 7. 页面影响

说明前端页面依赖哪些字段和接口。

### 8. 开发顺序

| 优先级 | 任务                   | 说明              |
| --- | -------------------- | --------------- |
| P0  | auth/me/login/logout | 前端关闭 mock 的前置条件 |
| P1  | 权限和租户隔离              | 所有业务接口前置条件      |

### 9. 待确认问题

| 问题 | 影响 | 默认方案 |
| -- | -- | ---- |

## 生成前自检

生成文档前必须检查：

* 是否存在可删的“目标”章节？
* 是否把普通列表错误写成代码块？
* 是否有超过 8 项的列表可以改成表格？
* 是否存在“建议”“相关”“等等”等模糊词？
* 是否有解释性段落可以压缩成一行规则？
* 是否保留了权限字符、状态枚举、接口路径等定义？
* 是否把 DTO / Schema 放入附录？
* 是否能让人类 5 分钟内审完主文档？
* 是否能让 Agent 根据接口、字段、规则直接编码？

---

## 输出要求

最终文档必须满足：

* 中文简洁。
* 定义完整。
* 规则明确。
* 结构稳定。
* 少解释。
* 少代码块。
* 少无意义列表。
* 多表格。
* 多契约。
* 人能快速审，Agent 能直接做。
